# 软件项目管理综合实验报告

## 一、项目背景与小组分工

### 1.1 项目背景

本项目为《软件项目管理》课程综合实验，以"校园活动报名与签到管理系统"为统一项目背景。该系统旨在为校园活动提供一站式管理平台，支持活动发布、学生报名、名单审核、二维码签到、签到统计和活动总结导出等基本功能。

本实验的重点并非完整功能实现，而是在统一项目情境下综合运用范围计划、进度计划与配置管理的相关知识，完成项目管理文档的编制与版本协作过程。

### 1.2 小组分工

| 姓名   | 角色       | 主要职责                         |
| ------ | ---------- | -------------------------------- |
| 葛团辉 | 项目经理   | 统筹进度、最终提交和冲突控制解决 |
| 马永超 | 计划负责人 | 负责 WBS 与进度计划文档          |
| 林立天 | 协作成员   | 参与文档完善、分支提交与冲突解决 |

### 1.3 协作约定

**提交命名规范**：每位成员使用本人姓名拼音全称作为 Git 提交人名称，确保提交记录可追溯。成员在向仓库提交内容时，Git 提交人名称统一设置为本人姓名的拼音全称。

**分支使用规则**：每位成员在自己的 feature 分支上开发，完成后合并到 dev 分支进行集成，最终由项目经理统一合并到 main 分支。

**合并责任人**：项目经理（葛团辉）负责最终合并到 main 分支。

**日常沟通方式**：通过线下讨论和即时通讯工具进行日常沟通协作。

### 2.2 三级分解逻辑

本 WBS 采用**以交付物为导向 + 按生命周期阶段**相结合的分解方式，具体逻辑如下：

1. **项目管理成果**：按照项目生命周期（启动→计划→配置→总结）纵向分解，确保每个阶段有明确的文档交付物，便于进度跟踪和过程管理。

2. **业务功能成果**：围绕系统三大核心业务模块（活动管理、报名管理、签到管理）横向分解，每个模块独立为二级节点，体现功能内聚性。模块之间存在依赖关系：活动管理是报名和签到的基础。

3. **技术实现成果**：按技术栈分层分解（数据库→前端→后端→测试），确保每个技术层次有具体的交付成果，层次清晰，便于分工协作。

4. **部署与提交成果**：作为收尾阶段，涵盖环境配置和版本控制，确保最终成果可运行、可提交。

**分解原则**：

坚持结果导向，不以"开会、讨论、修改代码、联调"等纯动作作为主要分解主线。

三级节点为最小可交付单元，每个节点均可独立验证和分配责任人。

同级条目之间遵循 MECE 原则（相互独立，完全穷尽）。

### 4.2 分支策略

```
main          ← 最终交付分支，受保护，只允许合并不允许直接推送
  └── dev     ← 集成分支，阶段性功能合并到此验证
       ├── feature-wbs        ← WBS 分解文档开发
       ├── feature-schedule   ← 进度计划开发
       ├── feature-config     ← 配置管理文档开发
       └── feature-report     ← 总结报告开发
```

- **main**：最终提交分支，保持稳定。所有开发完成后，由项目经理合并至此分支。
- **dev**：阶段集成分支，各 feature 分支完成后先合并到 dev 进行集成验证。
- **feature-\***：成员个人功能分支，以 `feature-` 为前缀，命名清晰反映功能内容。

### 4.3 提交与合并规则

1. **提交规范**：
   - Git 提交人名称统一设置为本人姓名的拼音全称（如葛团辉 → getuanhui）。
   - 提交信息采用简洁描述，说明本次提交的主要内容。
   - 每个功能点完成后及时提交，避免大量改动集中提交。

2. **合并规则**：
   - 各成员在个人 feature 分支完成开发后，先合并到 dev 分支。
   - dev 分支验证通过后，由项目经理（葛团辉）创建合并请求到 main 分支。
   - 合并时如遇冲突，由相关成员协商解决后提交。

3. **版本标记**：
   - 主要里程碑（如 WBS 完成、进度计划完成、最终提交）在提交信息中明确标注。

## 五、冲突制造、识别与解决过程

### 5.1 冲突制造

根据实验要求，我们通过以下方式制造了一次 Git 合并冲突：

**操作步骤**：

1. 基于提供的 `docs/conflict.md` 冲突示例文件进行操作。该文件的"当前结论"行为冲突触发点：
   ```
   当前结论：本组决定先合并到 dev，再由项目经理统一合并到 main。
   ```

2. 葛团辉在 `feature-wbs` 分支上将"当前结论"改为：
   ```
   当前结论：本组决定直接合并到 main。
   ```

3. 林立天在 `dev` 分支上（基于不同分支）将同一行的"当前结论"改为：
   ```
   当前结论：本组决定先经过 dev 再合并到 main。
   ```

4. 将两个分支进行合并时，Git 检测到同一文件同一位置被不同分支以不同方式修改，触发了合并冲突。

### 5.2 冲突识别

当执行 `git merge` 命令时，Git 输出以下冲突提示：
```
Auto-merging docs/conflict.md
CONFLICT (content): Merge conflict in docs/conflict.md
Automatic merge failed; fix conflicts and then commit the result.
```

使用 `git status` 查看，发现 `docs/conflict.md` 处于"both modified"（双方均修改）状态。打开文件后可以看到标准的冲突标记：

```diff
<<<<<<< HEAD
当前结论：本组决定直接合并到 main。
=======
当前结论：本组决定先经过 dev 再合并到 main。
>>>>>>> dev
```

### 5.3 冲突解决过程

1. **分析冲突原因**：两名成员在不同分支上修改了同一文件的同一行内容，Git 无法自动判断应该保留哪个版本。

2. **协商解决**：葛团辉（项目经理）与林立天通过讨论确定最终的合并方案。最终决定采用分阶段集成的策略，既保留 dev 分支的集成验证环节，又明确最终合并到 main 的流程。

3. **解决冲突**：手动编辑 `conflict.md` 文件，删除冲突标记，保留达成一致的最终结论：
   ```
   当前结论：本组决定各成员先在 feature 分支开发，完成后先合并到 dev 进行集成验证，再由项目经理统一合并到 main。
   ```

4. **提交解决**：使用 `git add` 标记冲突已解决，然后 `git commit` 完成合并提交，保留完整的冲突解决记录。

### 5.4 冲突解决总结

| 项目 | 内容 |
|------|------|
| 冲突文件 | `docs/conflict.md` |
| 冲突位置 | 第 5 行"当前结论" |
| 产生原因 | 两人在不同分支修改同一文件同一行 |
| 解决方式 | 协商确定最终方案，手动编辑解决 |
| 最终结果 | 保留完整提交记录，合并成功 |

## 六、实验总结

### 6.1 项目组织情况

本实验以3人小组形式开展，按照项目管理角色进行了明确分工：葛团辉担任项目经理统筹全局进度与冲突控制，马永超负责 WBS 与进度计划文档的编制，林立天负责配置管理与协助解决冲突。各成员均使用本人姓名拼音全称配置 Git 提交人信息，确保提交记录可追溯。

在协作过程中，我们采用了 feature 分支开发 → dev 分支集成 → main 分支交付的分层策略，保证了开发过程的规范性和可控性。项目经理负责最终的合并控制，确保项目交付质量。

### 6.2 WBS 分解体会

通过本次 WBS 分解实践，我们深刻理解了**以交付物为导向**的分解原则。WBS 的核心是回答"需要交付什么"，而非"需要做什么"。因此，我们的分解围绕业务功能块展开，而不是按照"开会、讨论、编码"等活动来划分。

三级分解使我们能够将项目从宏观到微观逐层细化，每一级都保持完整的交付物特征。这种结构化的分解方式不仅有助于明确工作范围，也为后续的进度计划编制和任务分配提供了清晰的基础。

### 6.3 实验收获

1. **Git 协作能力提升**：通过实际操作，掌握了仓库初始化、分支管理、合并操作、冲突处理等核心 Git 技能，尤其是在多人协作场景下的工作流程。

2. **项目管理实践**：将课程所学的范围管理（WBS）、时间管理（进度计划）和配置管理知识应用到具体实验中，理论与实践相结合，加深了对项目管理过程的理解。

3. **冲突解决经验**：通过主动制造并解决合并冲突，掌握了冲突识别方法（识别冲突标记）和冲突解决策略（协商→编辑→提交），认识到良好的分支策略可以有效减少冲突发生的频率和复杂度。

4. **团队协作意识**：体验了规范化的团队协作流程，包括统一提交规范、分支命名约定、合并流程等，认识到规范化的协作流程对于保证项目质量的重要性。

### 6.4 遇到的问题与改进

1. **问题**：初期分支命名使用了冒号（`feature:wbs`），但 Git 不允许分支名中包含冒号。
   **解决**：改用连字符分隔（`feature-wbs`），符合 Git 命名规范。

2. **问题**：远程仓库未提前配置，首次推送时遇到"没有推送目标"错误。
   **解决**：配置远程仓库地址后推送成功。

3. **改进建议**：后续可以引入 Pull Request / Merge Request 机制，在合并前进行代码审查，进一步提高协作质量。同时可以配合分支保护规则，防止误操作直接推送到 main 分支。
